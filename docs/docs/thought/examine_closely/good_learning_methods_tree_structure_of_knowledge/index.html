好的学习方式：知识点呈现为树结构

学习的步骤#

忘了那本书讲的，大概就是下面意思，过几天补

  1. 整理信息
  2. 建立知识点
  3. 关联知识点
  4. 应用知识点，建立稳固的概念

整理信息#

如何梳理知识#

根据以上学习的基本理论，我们要有好的信息收集机制。

不过我不知道别人怎么学习的，这里仅分享下自己最近掌握的、认为比较好用的梳理知识方法。

好的知识分类会呈现树状的数据结构#

这样设计的依据是，树状结构查询时比较快的。

假想场景：我们要查找一个知识点 x，它位于一堆 node 节点之中。

  1. 如果我们以 list 结构组织 node，有两个缺陷：
    1. 先学的在 list 底部，后学的在 list 顶部，而我们需要遍历 list 全部节点。
    2. list 长度有限。
      1. 前脚学后脚忘
      2. 捡了芝麻丢了西瓜
      3. 对不起我只能记 5 分钟，30 天前学过的东西，早就还给马克思了
// 悲剧了，时间久远的，x 是很早之前插入到 list 的，在 list 最底const list = [  'a',  'b',  ...,  'x',  'y',  'z']
// 悲剧了，脑容量太小，list 只能存 5 个 nodeconst list = [  'a',  'b',  'c',  'd',  'e',]
  1. 如果我们以 tree 结构组织知识。例如，只需要知道 Docker 位于 tree 数据结构的路径 Linux => 虚拟化 => Docker
const tree = {  Linux: {    虚拟化: {      Docker: {        ...一堆 Docker 相关的知识点      }    }  }}

树结构对我们提出更高的要求#

几点建议：

  • 编写文档时，将脑海中的知识点全部展开为标题。
  • 最上层节点是领域：例如 frontend, server, devops
  • 同级树节点之间的不能是包含关系。
    1. 例如 vue 和 react 下层组成元素虽然很接近，但属于两个不同的专题，因为对一般人的理解程度来说，它们在 API 方面的差距还是很大的。
    2. 前端下可以有 microapps 和 compile 等专题，它们不是互相包含的，只是有些交集
    3. microapps 和 qiankun。qiankun 就是 microapps 下的子节点，不能硬把他俩划分为同级
  • 同级树节点之间的关联性不能太疏远，也不能太过于密集。太疏远关联不到一起，太密集不好区别
  • 树的深度不能太高，否则可能查找路径太长

总的来说，就是呈现一个垂直方向深度不够、水平方向均匀分布的树

向树结构插入节点#

  • 插入一个节点时，要大概知道他属于什么领域，在树的什么位置
  • 大脑是有限的，我们只需要记住目录结构，而无需关注全部细节，借助文档隐藏大量细节
  • 重要信息全部整理起来，

真实的知识结构，不应该仅是树结构，更是网结构#

一个知识的组成，可能是多个知识点（俗称抓手），全方位的穿插（俗称赋能）在一起的（俗称组合拳）。

例如 Vue MVVM，大部分人就知道个生命周期 created, mounted 和响应式 Object.defineProperty()，其实背后有着诸多要素：

  1. 通过设计模式模板方法将生命周期插入 Vue 内部，在特定时机执行
  2. 通过观察者模式收集依赖
  3. 通过发布订阅关联视图层和模型层
  4. 编译时(vue-template-compiler)或运行时(Vue.compile) 将 template 解析为 render function
  5. 运行时执行 render function 生成 Virtual DOM
  6. DOM diff
  7. 还有很多，这里只列举我知道的……

例如，Docker 即是容器化技术，又便于发布部署

  1. 从运维的角度看它的路径，可以是 Linux => Continuous Delivery => Docker
  2. 从研发的角度看它的路径，也可以是 Linux => Containerization => Docker
