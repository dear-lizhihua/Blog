优化 web 应用程序

介绍#

对于 产品生命周期的不同阶段，我们进行不同的优化

编译时优化#

优化构建性能：构建工具#

构建工具自身的性能也值得优化

优化构建结果：分析应用程序中重复引用的组件#

通过 webpack analyze 分析，找到重复打包的模块，分析原因

优化构建结果：只保留引用的模块#

通过 tree shaking 进行优化，只保留模块中用到的部分

优化图片体积#

  1. 通过在编译时压缩图片质量，我们可以减少图片体积
  2. 为了避免重复构建，我们需要缓存构建结果

传输时优化#

优化构建结果：只传输按需加载的构建结果，只编译运行时用到的模块#

  1. 通过 Vite 等工具

开启 CDN 缓存#

  1. 通过 CDN 服务，加载最近服务上部署的静态资源

开启客户端缓存#

  1. 通过客户端缓存机制，我们将静态资源缓存在客户端中，避免下次访问重复请求相同的资源

开启 gzip 压缩#

优化 JSON 序列化性能(json serialization performance)#

通过 fast-json-stringify 等 JSON 序列化 library 来提高 JSON 序列化性能。

举例，fast-json-stringify 介绍如下：

  • 对于小负载，fast-json-stringify 比 JSON.stringify() 快得多。
  • 它的性能优势会随着负载的增长而缩小。
  • 它与 flatstr 搭配得很好，它会触发 V8 优化，从而在最终将字符串转换为 Buffer 时提高性能。

传输 protocol buffer 数据#

在 Beating JSON performance with Protobuf 这篇文章中提到：

Protobuf 是 Google 建立的二进制格式，即使在 Node.js/V8 和 web 浏览器等 JavaScript 环境中，其性能也超过了 JSON。

传输 buffer 数据#

在 Sending and Receiving Binary Data 这篇文章中提到：

可以设置 XMLHttpRequest 对象的 responseType 属性来更改来自 server 的预期响应类型。 可能的值为 "" 空字符串（默认）, "arraybuffer", "blob", "document", "json", "text"。response 属性将根据 responseType 包含实体主体，如 ArrayBuffer, Blob, Document, JSON, string。如果请求未完成或未成功，则为 null。

减少过多的请求数量：将体积小的 img url 转为 base64 url#

引擎加载时优化#

  1. reflow
  2. repaint

运行时优化#

  1. 使用 DOM diff 算法。通过 key 来复用组件
  2. 通过 service worker，开启多线程计算

用户交互优化#

  1. 数据结构优化。设计高效的数据结构，避免低效数据结构造成过多的内存占用
  2. 大量节点的树。通过虚拟滚动进行优化
  3. 大量行的表格。通过虚拟滚动进行优化
  4. 在用户切换路由时，我们缓存上个路由的组件状态
  5. 在频繁切换组件内部状态引发组件多次更新时，我们减少组件内部状态影响范围
