什么是负载均衡？

原文：What is Load Balancing?

介绍#

负载均衡是高可用基础设施的关键组件，通常是通过在多台 server 之间分发工作量，来提高网站、应用程序、数据库和其他服务的性能和可靠性。

一个没有负载均衡的 web 基础设施，可能类似如下：

在这个示例中，用户直接连接到 yourdomain.com 对应的 web server。如果该单个 web server 出现故障，用户就无法再访问此网站。此外，如果许多用户尝试在同一时间段内访问 server 而无法处理负载，他们可能会感受到较长的负载时间，或者根本无法连接。

通过在后端引入负载均衡服务，以及至少一个 web server，可以缓解这类单点故障。通常所有后端 server 都提供相同的内容，所以不论是由哪个服务器响应，用户都会收到一致的内容。

在上面的示例中，用户访问负载均衡服务，此负载均衡服务将用户的请求转发到后端 server，然后由后端 server 直接响应用户的请求。在这种场景中，负载均衡器现在自身都会出现单点故障。可以通过引入第二个负载均衡服务来缓解这种情况，但是在我们进行论述多个负载均衡服务之前，让我们先探讨一下负载均衡服务是如何工作的。

负载均衡服务可以处理哪些类型的流量？#

负载均衡服务的管理人员，会为四种主要类型的流量创建转发规则：

  • HTTP — 基于标准 HTTP 机制，通过标准的 HTTP 均衡来定向请求。负载均衡服务通过设置 X-Forwarded-For, X-Forwarded-Proto, X-Forwarded-Port headers，来向其后端提供原始请求信息。
  • HTTPS — HTTPS 均衡的功能与 HTTP 均衡相同，但增加了加密功能。加密是通过 SSL passthrough（始终保持内容加密，将加密的流量发送到后端）或 SSL termination（将解密工作交给负载均衡服务，然后把未加密的流量发送到后端）这两种方式中的一种进行处理。
  • TCP — 对于那些不使用 HTTP 或 HTTPS 的应用程序而言，还可以实现 TCP 流量的均衡。例如，去往数据库集群的流量，可以散播到所有 server 中。
  • UDP — 最近，一些负载均衡服务增加了对于负载均衡核心网络协议（如 DNS）和使用 UDP 的 syslogd 的支持。

这些转发规则会在负载均衡服务自身内部定义协议和端口，然后在转发规则与协议和端口之间建立映射，然后负载均衡服务会从流量中解析出协议和端口，并根据协议和端口将其映射到相应转发规则，最后根据转发规则把流量路由到后端。

负载均衡服务如何选择后端 server？#

负载均衡服务基于两个因素的组合，来选择将请求转发到哪个 server。负载均衡服务首先要确保所有可以选择的 server，实际上都在正常地响应请求，然后使用预先配置的规则，从这些健康运行 server 池中进行选择。

健康检查#

负载均衡服务应该仅将流量转发到状态是“健康”的后端 server。为了监视后端 server 的健康情况，健康检查会定期尝试使用转发规则定义的协议和端口连接到后端 server，以确保 server 正在监听请求。如果 server 无法通过健康检查，就表示无法 serve 请求，负载均衡服务会自动将其从池中删除，等到此 server 再次响应健康检查，流量才会转发过去。

负载均衡算法#

负载均衡算法，主要用于选择后端所有处于健康状态的 server。一些经常用到的算法是：

Round Robin — Round Robin 表示按顺序地选择 server。对于第一个请求，负载均衡服务在 server 列表中选择第一个 server 处理，然后按顺序地向下遍历列表，到达列尾时再回到列头。

Least Connections — Least Connection 表示负载均衡服务将选择连接最少的 server。当流量连接数过多导致会话时间较长时，推荐使用此算法。

Source — 使用 Source 算法，负载均衡服务根据一个基于源 IP 的哈希映射表（例如访问者的 IP 地址）来判断请求需要使用哪个 server 处理。这种方式可以确保某些特定的用户都一致地连接到同一 server。

实现不同的目的，需要使用特定的负载均衡技术，管理人员为此会用到不同的算法。

负载均衡服务如何处理状态？#

某些应用程序要求，用户持续地连接到同一个后端 server。一种方式是使用 Source 算法，根据客户端 IP 信息建立相似性。而在 web 应用程序层级，实现此需求的另一种方式是通过粘性会话(sticky session)，由负载均衡服务设置一个 cookie，并将来自此会话的所有请求，定向到同一个物理 server。

冗余负载均衡服务#

为了避免部署单点负载均衡服务时出现的故障，可以将第一个负载均衡服务和第二个负载均衡服务连接起来组成一个集群，每个服务都在其中监视另一个服务的健康情况。每个服务都具有相同的故障检测和故障恢复的能力。

如果主负载均衡服务发生故障，DNS 必须将用户请求发送到第二个负载均衡服务。由于 DNS 变更需要在 Internet 上传播，这会耗费大量时间。为了使此故障转移自动进行，许多管理人员会使用允许“flexible IP(弹性 IP) 地址重新映射（例如 floating IP(浮动 IP)）”的系统。按需 IP 地址重新映射通过提供可以在需要时重新映射的静态 IP 地址，消除了 DNS 变更固有的传播和缓存问题。域名可以保持与同一 IP 地址关联，而 IP 地址本身在 server 之间移动。
