谷歌云培训 - Cloud IAM 自定义角色（中文）

重要声明#

这篇使用 DeepL 翻译，这里访问 英文版本。

概述#

Cloud IAM 提供了正确的工具，以最小的麻烦和高度自动化来管理资源权限。你不会直接授予用户 permission。相反，你授予他们 role，它绑定了一个或多个 permission。这允许你将公司内的工作职能映射到组和角色。用户只获得他们完成工作所需的 permission，管理员可以轻松地将默认权限授予整个用户组。

在 Cloud IAM 中，有两种类型的角色。

  • 预定义角色
  • 自定义角色

预定义角色是由谷歌创建和维护的。他们的 permission 会在必要时自动更新，例如，当新的功能或服务被添加到 Google Cloud 时。

自定义角色是用户定义的，允许你绑定一个或多个支持的 permission 以满足你的特定需求。自定义角色不由谷歌维护；当新的权限、功能或服务被添加到 Google Cloud 时，您的自定义角色将不会自动更新。您通过组合一个或多个可用的 Cloud IAM 权限来创建一个自定义角色。 permission 允许用户在 Google Cloud 资源上执行特定的操作。

你要做什么#

  • 创建、更新、删除、撤销自定义角色

先决条件#

  • 熟悉 IAM 角色

设置#

在你点击 "开始实验室" 按钮之前#

请阅读这些说明。实验室是有时间限制的，你不能暂停它们。当你点击开始实验室时，计时器开始显示 Google Cloud 资源将为你提供多长时间。

这个 Qwiklabs 实践实验室让你在真实的云环境中，而不是在模拟或演示环境中，亲自进行实验活动。它通过给你新的、临时的凭证，让你在实验期间用来登录和访问 Google Cloud。

你需要什么#

要完成这个实验室，你需要。

  • 访问一个标准的互联网浏览器（推荐使用 Chrome 浏览器）。
  • 完成本实验的时间。

注意：如果你已经有了自己的个人 Google Cloud 账户或项目，请不要将其用于本实验。

注意：如果你使用的是 Pixelbook，请打开一个隐身窗口来运行本实验室。

    重要的是： 你必须使用 "连接详情 "面板上的证书。不要使用你的Qwiklabs凭证。如果你有自己的Google Cloud账户，不要在这个实验室中使用它（避免产生费用）。

了解 IAM 自定义角色#

Cloud IAM 还提供了创建自定义 Cloud IAM 角色的能力。您可以创建一个具有一个或多个权限的自定义 Cloud IAM 角色，然后将该自定义角色授予用户。Cloud IAM 提供了一个 UI 和 API，用于创建和管理自定义角色。

关键点：自定义角色使您能够执行最小特权原则，确保您组织中的用户和服务账户只拥有执行其预期功能所必需的权限。

注意：你可以在组织层面和项目层面创建一个自定义角色。但是，你不能在文件夹级别创建自定义角色。

您可以通过结合一个或多个可用的 Cloud IAM 权限来创建自定义角色。权限允许用户在 Google Cloud 资源上执行特定操作。

在 Cloud IAM 的世界里，权限是以这样的形式表示的。

<service>.<resource>.<verb>

例如，compute.instances.list权限允许用户列出他们拥有的 Compute Engine 实例，而 compute.instances.stop 允许用户停止一个虚拟机。

权限通常，但不总是，与 REST 方法 1:1 对应。也就是说，每个 Google Cloud 服务的每个 REST 方法都有一个相关的权限。要调用一个方法，调用者需要该权限。例如，topic.publish() 的调用者需要 pubsub.themes.publish 许可。

自定义角色只能用于在策略中为拥有其下的角色或资源的同一个项目或组织授予权限。你不能将一个项目或组织的自定义角色授予不同项目或组织拥有的资源。

所需的权限和角色#

要创建一个自定义角色，调用者必须有 iam.roles.create 权限。

不是所有者的用户，包括组织管理员，必须被分配到 Organization Role Administrator 角色（roles/iam.organizationRoleAdmin）或 IAM Role Administrator 角色（roles/iam.roleAdmin）。IAM Security Reviewer 角色（roles/iam.securityReviewer）能够查看自定义角色，但不能管理它们。

自定义角色的用户界面在 Cloud Console 的 IAM 角色下。它只对有权限创建或管理自定义角色的用户可用。默认情况下，只有项目所有者可以创建新的角色。项目所有者可以通过授予同一项目的其他人 IAM 角色管理员角色来控制对该功能的访问；对于组织来说，只有组织管理员可以授予组织角色、管理员角色。

在创建自定义角色之前，你可能想知道哪些权限可以应用于资源。您可以使用 gcloud 命令行工具、Cloud Console或 IAM API 获得可应用于某一资源的所有权限，以及该资源下面的层次结构。例如，您可以获得您可以应用在一个组织和该组织中的项目上的所有权限。

创建一个自定义角色#

要创建一个自定义角色，调用者必须拥有 iam.role.create 权限。默认情况下，项目或组织的所有者拥有这个权限，可以创建和管理自定义角色。

非所有者的用户，包括组织管理员，必须被分配到组织角色管理员的角色，或IAM角色管理员的角色。

使用 gcloud iam roles create 命令来创建新的自定义角色。您可以通过两种方式使用此命令。

  • 通过提供一个包含角色定义的YAML文件
  • 通过使用标志来指定角色定义 创建自定义角色时，您必须通过使用 --organization [ORGANIZATION_ID] 或 --project [PROJECT_ID] 标志来指定它是适用于组织级别还是项目级别。下面的每个例子都在项目级别创建了一个自定义角色。

在接下来的练习中，你将在项目级别上创建自定义角色。

使用 YAML 文件来创建一个自定义角色#

创建一个 YAML 文件，其中包含自定义角色的定义。该文件的结构必须如下。

title: [ROLE_TITLE]description: [ROLE_DESCRIPTION]stage: [LAUNCH_STAGE]includedPermissions:- [PERMISSION_1]- [PERMISSION_2]

每个占位符的值描述如下。

  • [ROLE_TITLE]是角色的友好标题，如角色查看器。
  • [ROLE_DESCRIPTION]是关于该角色的简短描述，如我的自定义角色描述。
  • [LAUNCH_STAGE]表示角色在启动生命周期中的阶段，如 ALPHA、BETA 或 GA。
  • includedPermissions指定在自定义角色中包含的一个或多个权限的列表，如iam.roles.get。

是时候开始了! 通过运行创建你的角色定义 YAML 文件。

nano role-definition.yaml

在 YAML 文件中添加这个自定义角色定义。

title: "Role Editor"description: "Edit access for App Versions"stage: "ALPHA"includedPermissions:- appengine.versions.create- appengine.versions.delete

然后保存并关闭该文件。

执行以下 gcloud 命令。

gcloud iam roles create editor --project $DEVSHELL_PROJECT_ID \--file role-definition.yaml

如果角色创建成功，将返回以下响应。

Created role [editor].description: Edit access for App Versionsetag: BwVs4O4E3e4=includedPermissions:- appengine.versions.create- appengine.versions.deletename: projects/[PROJECT_ID]/roles/editorstage: ALPHAtitle: Role Editor

使用标志创建一个自定义角色#

现在你将使用 flag 方法来创建一个新的自定义角色。标志的形式与 YAML 文件类似，所以你会认识到该命令是如何建立的。

执行下面的 gcloud 命令，使用 flags 创建一个新角色。

gcloud iam roles create viewer --project $DEVSHELL_PROJECT_ID \--title "Role Viewer" --description "Custom role description." \--permissions compute.instances.get,compute.instances.list --stage ALPHA

编辑一个现有的自定义角色#

更新资源元数据（比如自定义角色）的一个常见模式是读取其当前状态，在本地更新数据，然后发送修改后的数据进行写入。如果两个或多个独立的进程同时尝试这个序列，这种模式可能会导致冲突。

例如，如果一个项目的两个所有者试图同时对一个角色进行冲突的修改，一些修改可能会失败。

Cloud IAM 使用自定义角色中的 etag 属性来解决这个问题。该属性用于验证自定义角色自上次请求以来是否有变化。当您向 Cloud IAM 提出带有 etag 值的请求时，Cloud IAM 会将请求中的 etag 值与自定义角色相关的现有 etag 值进行比较。只有当 etag 值匹配时，它才会写入更改。

使用 gcloud iam roles update 命令来更新自定义角色。您可以通过两种方式使用此命令。

  • 通过提供一个包含更新的角色定义的 YAML 文件
  • 通过使用标志来指定更新的角色定义

当更新自定义角色时，您必须通过使用 --organization [ORGANIZATION_ID] 或 --project [PROJECT_ID] 标志来指定它是适用于组织级别还是项目级别。下面的每个例子都在项目级创建了一个自定义角色。

describe 命令返回角色的定义，并包括一个 etag 值，用来唯一地识别角色的当前版本。这个 etag 值应该在更新的角色定义中提供，以确保任何同时进行的角色修改不会被覆盖。

使用 YAML 文件更新一个自定义角色#

通过执行以下 gcloud 命令获得角色的当前定义，用editor替换 [ROLE_ID]。

gcloud iam roles describe [ROLE_ID] --project $DEVSHELL_PROJECT_ID

describe 命令返回以下输出。

description: [ROLE_DESCRIPTION]etag: [ETAG_VALUE]includedPermissions:- [PERMISSION_1]- [PERMISSION_2]name: [ROLE_ID]stage: [LAUNCH_STAGE]title: [ROLE_TITLE]

你将用输出的值创建一个新的 YAML 文件。复制这个命令的输出。

用你的编辑器创建一个 new-role-definition.yaml 文件。

nano new-role-definition.yaml

粘贴上一条命令的输出值，在 includedPermissions 下添加这两个权限：

- storage.buckets.get- storage.buckets.list

当你完成后，你的 YAML 文件将看起来像这样：

description: Edit access for App Versionsetag: BwVxIAbRq_I=includedPermissions:- appengine.versions.create- appengine.versions.delete- storage.buckets.get- storage.buckets.listname: projects/[PROJECT_ID]/roles/editorstage: ALPHAtitle: Role Editor

保存并关闭该文件。

现在你将使用 update 命令来更新这个角色。执行下面的 gcloud 命令，将 [ROLE_ID] 替换为 editor：

gcloud iam roles update [ROLE_ID] --project $DEVSHELL_PROJECT_ID \--file new-role-definition.yaml

如果角色被成功更新，将返回以下响应：

description: Edit access for App Versionsetag: BwVxIBjfN3M=includedPermissions:- appengine.versions.create- appengine.versions.delete- storage.buckets.get- storage.buckets.listname: projects/[PROJECT_ID]/roles/editorstage: ALPHAtitle: Role Editor

使用标志来更新一个自定义的角色#

角色定义的每个部分都可以使用相应的标志来更新。请参阅 gcloud iam roles update 主题，了解所有可能的标志的列表。

您可以使用以下标志来添加或删除权限。

  • --add-permissions: 向角色添加一个或多个用逗号分隔的权限。
  • --remove-permissions: 从角色中删除一个或多个逗号分隔的权限。

或者，你可以使用 --permissions [PERMISSIONS] 标志来指定新的权限，并提供一个逗号分隔的权限列表来替换现有的权限列表。

执行下面的 gcloud 命令，使用标志向 viewer 角色添加权限。

gcloud iam roles update viewer --project $DEVSHELL_PROJECT_ID \--add-permissions storage.buckets.get,storage.buckets.list

如果角色被成功更新，将返回以下响应：

description: Custom role description.etag: BwVxLi4wTvk=includedPermissions:- compute.instances.get- compute.instances.list- storage.buckets.get- storage.buckets.listname: projects/[PROJECT_ID]/roles/viewerstage: ALPHAtitle: Role Viewer

禁用一个自定义角色#

当一个角色被禁用时，任何与该角色相关的策略绑定都会被激活，这意味着该角色中的权限将不会被授予，即使你将该角色授予一个用户。

停用现有自定义角色的最简单方法是使用 --stage flag 并将其设置为 DISABLED。

执行下面的 gcloud 命令来禁用 viewer 角色。

gcloud iam roles update viewer --project $DEVSHELL_PROJECT_ID \--stage DISABLED

如果角色被成功更新，将返回以下响应。

description: Custom role description.etag: BwVxLkIYHrQ=includedPermissions:- compute.instances.get- compute.instances.list- storage.buckets.get- storage.buckets.listname: projects/[PROJECT_ID]/roles/viewerstage: DISABLEDtitle: Role Viewer

点击检查我的进度来验证目标。停用一个自定义角色

删除一个自定义角色#

使用 gcloud iam roles delete 命令来删除自定义角色。一旦删除，该角色将失去作用，不能用于创建新的 IAM 策略绑定。

gcloud iam roles delete viewer --project $DEVSHELL_PROJECT_ID

Example Output:

description: Custom role description.etag: BwVxLkf_epw=includedPermissions:- compute.instances.get- compute.instances.list- storage.buckets.get- storage.buckets.listname: projects/[PROJECT_ID]/roles/viewerstage: DISABLEDtitle: Role Viewer

角色被删除后，现有的绑定仍然存在，但不活跃。该角色可以在 7 天内撤销删除。7 天后，该角色进入一个永久删除过程，持续 30 天。37 天后，角色 ID 可以再次被使用。

注意：如果一个角色被淘汰，将其 role.stage 属性改为 DEPRECATED，并设置 deprecation_message 以让用户知道应该使用什么替代角色或在哪里获得更多信息。

撤销一个自定义角色#

在 7 天的时间内，你可以撤销一个角色。被删除的角色处于 DISABLED 状态。你可以通过更新 --stage flag 使其再次可用。

gcloud iam roles undelete viewer --project $DEVSHELL_PROJECT_ID
